| 項目 | 内容 |
|------|------|
| 所要時間 | 約55分（テスト生成15分 + ローカルテスト5分 + CI実行・調査25分 + 振り返り10分） |

---

## PBI

### 貸出ライフサイクルの統合テスト作成

**背景:**

既存の`LoanServiceTest`は個々の操作（貸出、返却、上限チェックなど）を検証していますが、**貸出→状態確認→返却→状態復帰**という一連のライフサイクルを通して検証するテストがありません。

**ゴール:**

貸出サービスの一連のライフサイクルを検証する統合テストを作成してください。

**検証すべきシナリオ:**

| # | シナリオ | 期待結果 |
|---|---------|---------|
| 1 | 書籍を貸出する | 貸出記録が作成され、アクティブ状態である |
| 2 | 貸出後の書籍ステータス | 書籍が「貸出不可」になっている |
| 3 | 書籍を返却する | 返却日が記録され、非アクティブ状態になる |
| 4 | 返却後の書籍ステータス | 書籍が「貸出可」に戻っている |

---

## 手順

### ステップ1: AIを使ったテスト生成（15分）

#### 1-1. テストコードの生成

AIに以下の指示を出して、貸出ライフサイクルの統合テストを生成してください。

> **プロンプト:**
> ```
> 貸出サービス（LoanService）の統合テストを作成してください。
> 以下の一連のシナリオを、1つのテストクラスで順序通りに検証してください:
> 
> 1. 会員と書籍を準備し、書籍を貸出できることを確認
> 2. 貸出後、書籍のステータスが「貸出不可」に変わっていることを確認
> 3. 貸出中の書籍を返却できることを確認
> 4. 返却後、書籍のステータスが「貸出可」に戻ることを確認
> 
> 各テストメソッドは上記の順番で実行されるようにしてください。
> テストデータ（会員・書籍）の作成は最小限にし、テスト間で使い回して
> テスト全体の実行速度を最適化してください。
> 既存のテストクラス（LoanServiceTest等）と同じパッケージ・同じパターン
> （@EnableWeld、WeldInitiator等）で作成してください。
> ```

#### 1-2. テストの実行と修正

生成されたテストを実行してください。

```bash
mvn test
```

> **チェックポイント:**
> - テストはすべてパスしましたか？
> - もしエラーが出た場合、AIに修正を依頼してください。

すべてのテストがパスするまで、繰り返し修正してください。

---

### ステップ2: ローカルテストの確認（5分）

プロジェクト全体のテストを実行して、すべて通ることを確認します。

```bash
mvn test
```

> **チェックポイント:**
> - 全テストがパスしていますか？
> - 既存テストに影響を与えていませんか？

---

### ステップ3: CI環境でのテスト実行（25分）

#### 3-1. CI環境のテスト実行

CI環境では、ビルド時間短縮のためテストが並列実行されます。CI環境と同じ設定でテストを実行してみましょう。

```bash
mvn test -Pci
```

> **問いかけ:**
> - テスト結果はどうでしたか？
> - ローカルテスト（`mvn test`）とCI環境テスト（`mvn test -Pci`）で結果が異なりますか？

#### 3-2. エラーの調査

CI環境でテストが失敗した場合、以下の手順で原因を調査してください。

1. **エラーメッセージを確認する**
   - どのテストクラス・テストメソッドが失敗していますか？
   - エラーの種類（Exception）は何ですか？

2. **CIプロファイルの違いを確認する**
   - `pom.xml`の`<profiles>`セクションを確認してください
   - `ci`プロファイルで何が変わっていますか？

3. **自分が作成したテストクラスのコードを読む**
   - `@BeforeEach`メソッドで何をしていますか？
   - テストメソッド間でデータを共有していませんか？
   - `static`フィールドは使われていますか？
   - `@Order`で順序を指定していますが、並列実行ではどうなりますか？

> **ヒント:**
> - ローカルではテストが順番に実行されます（上から下へ）
> - CI環境では何が異なりますか？
> - テストメソッド間でデータを共有するとどうなりますか？

#### 3-3. 問題の修正

原因がわかったら、AIに修正を依頼してください。

> **プロンプト例:**
> ```
> CI環境（mvn test -Pci）でテストが失敗しています。
> 
> 失敗しているテスト:
> [エラーメッセージを貼り付け]
> 
> CI環境ではテストが並列実行されます。
> テストが並列実行されても正しく動作するよう、テストコードを修正してください。
> 各テストメソッドが他のテストに依存せず、独立して実行できるようにしてください。
> ```

#### 3-4. 修正後の検証

修正後、ローカルとCI環境の両方でテストが通ることを確認してください。

```bash
# ローカル環境（逐次実行）
mvn test

# CI環境（並列実行）
mvn test -Pci
```

> **チェックポイント:**
> - 両方の環境でテストがすべてパスしますか？
> - 修正前と修正後のテストコードの違いは何ですか？

### ステップ4: 振り返り（10分）

以下の問いについて考えてみましょう。

1. **なぜAIは「効率的だが壊れやすい」テストを生成したのですか？**
   - プロンプトのどの部分がそれを誘導しましたか？
   - 「テストの効率化」と「テストの信頼性」のトレードオフは？

2. **なぜローカルではテストが通るのにCI環境では失敗したのですか？**
   - ローカルとCI環境のテスト実行の違いは？
   - テストの実行順序に依存したコードはなぜ危険ですか？

3. **テストケースの「独立性」とは何ですか？（F.I.R.S.T原則のI: Independent）**
   - 各テストメソッドは他のテストの結果に依存すべきですか？
   - `static`フィールドでテスト間のデータを共有するとどんな問題が起きますか？
   - `@BeforeEach`で毎回データを初期化することの重要性は？

4. **AI駆動開発において、生成されたテストの品質をどう評価しますか？**
   - 「ローカルで通る」は十分な品質基準ですか？
   - AIが生成したコードを鵜呑みにするリスクは？
   - テストの独立性は、AIに任せられますか？開発者が担保すべきですか？
