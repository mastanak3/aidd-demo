## PBI

### 蔵書検索機能の追加

**ユーザーストーリー:**

> 図書館の利用者として、タイトルや著者名で書籍を検索したい。そうすることで、読みたい書籍をすばやく見つけることができる。

**Acceptance Criteria:**

| # | 条件 | 詳細 |
|---|------|------|
| AC-1 | タイトルで検索 | タイトルのキーワードで書籍を部分一致検索できる |
| AC-2 | 著者名で検索 | 著者名のキーワードで書籍を部分一致検索できる |
| AC-3 | 貸出可能な書籍の絞り込み | 貸出可能な書籍のみをフィルタリングできる |
| AC-4 | 検索結果0件 | 該当する書籍がない場合、空のリストを返す |
| AC-5 | キーワード空 | キーワードが空の場合、全書籍を返す |

**検索例:**

| 検索条件 | 登録書籍 | 検索結果 |
|----------|----------|----------|
| タイトル「テスト」 | 「テスト駆動開発」「リファクタリング」 | 「テスト駆動開発」のみ |
| 著者「Beck」 | 「テスト駆動開発(Kent Beck)」「リファクタリング(Martin Fowler)」 | 「テスト駆動開発」のみ |
| タイトル「存在しない」 | 任意 | 空リスト |

### ステップ3: CI環境でのテスト実行（20分）

#### 3-1. CI環境のテスト実行

CI環境では、ビルド時間短縮のためテストが並列実行されます。CI環境と同じ設定でテストを実行してみましょう。

```bash
mvn test -Pci
```

> **問いかけ:**
> - テスト結果はどうでしたか？
> - ローカルテスト（`mvn test`）とCI環境テスト（`mvn test -Pci`）で結果が異なりますか？

#### 3-2. エラーの調査

CI環境でテストが失敗した場合、以下の手順で原因を調査してください。

1. **エラーメッセージを確認する**
   - どのテストクラス・テストメソッドが失敗していますか？
   - エラーの種類（Exception）は何ですか？

2. **CIプロファイルの違いを確認する**
   - `pom.xml`の`<profiles>`セクションを確認してください
   - `ci`プロファイルで何が変わっていますか？

3. **失敗するテストクラスのコードを読む**
   - 失敗しているテストクラスのソースコードを開いてください
   - `@BeforeEach`メソッドで何をしていますか？
   - テストメソッド間でデータを共有していませんか？
   - `static`フィールドは使われていますか？

> **ヒント:**
> - ローカルではテストが順番に実行されます（上から下へ）
> - CI環境では何が異なりますか？
> - テストメソッド間でデータを共有するとどうなりますか？

#### 3-3. 問題の修正

原因がわかったら、AIに修正を依頼してください。

> **プロンプト例:**
> ```
> CI環境（mvn test -Pci）でテストが失敗しています。
> 
> 失敗しているテスト:
> [エラーメッセージを貼り付け]
> 
> CI環境ではテストが並列実行されます。
> テストが並列実行されても正しく動作するよう、テストコードを修正してください。
> 各テストメソッドが他のテストに依存せず、独立して実行できるようにしてください。
> ```

#### 3-4. 修正後の検証

修正後、ローカルとCI環境の両方でテストが通ることを確認してください。

```bash
# ローカル環境（逐次実行）
mvn test

# CI環境（並列実行）
mvn test -Pci
```

> **チェックポイント:**
> - 両方の環境でテストがすべてパスしますか？
> - 修正前と修正後のテストコードの違いは何ですか？

### ステップ4: 振り返り（15分）

以下の問いについて考えてみましょう。

1. **なぜローカルではテストが通るのにCI環境では失敗したのですか？**
   - ローカルとCI環境のテスト実行の違いは？
   - テストの実行順序に依存したコードはなぜ危険ですか？

2. **テストケースの「独立性」とは何ですか？**
   - 各テストメソッドは他のテストメソッドの結果に依存すべきですか？
   - `static`フィールドでテスト間のデータを共有するとどんな問題が起きますか？
   - `@BeforeEach`で毎回データを初期化することの重要性は？

3. **F.I.R.S.T原則の「I（Independent）」について**
   - テストが独立であるとはどういうことですか？
   - テストの独立性を確保するための具体的な方法は？
   - Setup/Teardownはなぜ重要ですか？

4. **「ローカルで動いたからOK」は十分ですか？**
   - ローカル環境とCI環境の違いをどう考慮すべきですか？
   - テストの信頼性を高めるためにどんな工夫ができますか？

5. **AI駆動開発において、テストの独立性をどう担保しますか？**
   - AIが生成したテストコードの品質をどう評価しますか？
   - テスト間の依存関係をどう発見しますか？
   - CI環境でのテスト実行を前提としたテスト設計とは？
