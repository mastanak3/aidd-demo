# 課題4: 貸出更新機能の実装と統合テスト

## 課題概要

| 項目 | 内容 |
|------|------|
| 所要時間 | 約60分（PBI実装10分 + テスト生成10分 + ローカルテスト5分 + CI実行・調査25分 + 振り返り10分） |
| 前提知識 | ベースラインプロジェクトでの開発経験 |

---

## PBI

### 貸出更新（リニューアル）機能の追加

**ユーザーストーリー:**

> 図書館の利用者として、借りている本の返却期限を延長したい。そうすることで、まだ読み終わっていない本をもう少し長く借りられる。

**Acceptance Criteria:**

| # | 条件 | 期待結果 |
|---|------|---------|
| AC-1 | アクティブな貸出を更新する | 返却期限が会員種別の貸出期間分、現在の日付から再計算される |
| AC-2 | 既に返却済みの貸出を更新しようとする | エラーになる |
| AC-3 | 更新後も書籍のステータスは変わらない | 書籍が「貸出不可」のまま |

---

## 手順

### ステップ1: PBI実装（10分）

#### 1-1. プロダクトコードの変更

AIに以下の指示を出して、貸出更新機能を実装してください。

> **プロンプト例:**
> ```
> 以下のPBIに基づいて、貸出更新（リニューアル）機能を実装してください。
> 
> ユーザーストーリー:
> 図書館の利用者として、借りている本の返却期限を延長したい。
> 
> Acceptance Criteria:
> - アクティブな貸出を更新すると、返却期限が会員種別の貸出期間分、現在の日付から再計算される
> - 既に返却済みの貸出を更新しようとするとエラーになる
> - 更新後も書籍のステータスは貸出中のまま
> 
> LoanServiceにrenewLoanメソッドを追加してください。
> Loanドメインモデルにも必要な変更を加えてください。
> テストコードはまだ変更しないでください。
> ```

#### 1-2. 既存テストの実行

変更後、既存テストを実行して壊れていないことを確認してください。

```bash
mvn test
```

> **確認ポイント:**
> - 既存テストはすべてパスしましたか？
> - 新しいコードは既存のアーキテクチャパターン（サービス層、ドメインモデル）に沿っていますか？

---

### ステップ2: AIを使ったテスト生成（10分）

#### 2-1. テストコードの生成

AIに以下の指示を出して、貸出更新機能を含むライフサイクルの統合テストを生成してください。

> **プロンプト:**
> ```
> 貸出サービス（LoanService）の統合テストを作成してください。
> 貸出更新（renewLoan）を含む一連のライフサイクルを、
> 1つのテストクラスで順序通りに検証してください:
> 
> 1. 会員と書籍を準備し、書籍を貸出できることを確認
> 2. 貸出を更新して、返却期限が延長されることを確認
> 3. 更新後、書籍のステータスが「貸出不可」のままであることを確認
> 4. 書籍を返却して、ステータスが「貸出可」に戻ることを確認
> 
> 各テストメソッドは上記の順番で実行されるようにしてください。
> テストデータ（会員・書籍）の作成は最小限にし、テスト間で使い回して
> テスト全体の実行速度を最適化してください。
> 既存のテストクラス（LoanServiceTest等）と同じパッケージ・同じパターン
> （@EnableWeld、WeldInitiator等）で作成してください。
> ```

#### 2-2. テストの実行と修正

生成されたテストを実行してください。

```bash
mvn test
```

> **チェックポイント:**
> - テストはすべてパスしましたか？
> - もしエラーが出た場合、AIに修正を依頼してください。

すべてのテストがパスするまで、繰り返し修正してください。

### ステップ4: CI環境でのテスト実行（25分）

#### 4-1. CI環境のテスト実行

CI環境では、ビルド時間短縮のためテストが並列実行されます。CI環境と同じ設定でテストを実行してみましょう。

```bash
mvn test -Pci
```

> **問いかけ:**
> - テスト結果はどうでしたか？
> - ローカルテスト（`mvn test`）とCI環境テスト（`mvn test -Pci`）で結果が異なりますか？

#### 4-2. エラーの調査

CI環境でテストが失敗した場合、以下の手順で原因を調査してください。

1. **エラーメッセージを確認する**
   - どのテストクラス・テストメソッドが失敗していますか？
   - エラーの種類（Exception）は何ですか？

2. **CIプロファイルの違いを確認する**
   - `pom.xml`の`<profiles>`セクションを確認してください
   - `ci`プロファイルで何が変わっていますか？

3. **自分が作成したテストクラスのコードを読む**
   - `@BeforeEach`メソッドで何をしていますか？
   - テストメソッド間でデータを共有していませんか？
   - `static`フィールドは使われていますか？
   - `@Order`で順序を指定していますが、並列実行ではどうなりますか？

> **ヒント:**
> - ローカルではテストが順番に実行されます（上から下へ）
> - CI環境では何が異なりますか？
> - テストメソッド間でデータを共有するとどうなりますか？

#### 4-3. 問題の修正

原因がわかったら、AIに修正を依頼してください。

> **プロンプト例:**
> ```
> CI環境（mvn test -Pci）でテストが失敗しています。
> 
> 失敗しているテスト:
> [エラーメッセージを貼り付け]
> 
> CI環境ではテストが並列実行されます。
> テストが並列実行されても正しく動作するよう、テストコードを修正してください。
> 各テストメソッドが他のテストに依存せず、独立して実行できるようにしてください。
> ```

#### 4-4. 修正後の検証

修正後、ローカルとCI環境の両方でテストが通ることを確認してください。

```bash
# ローカル環境（逐次実行）
mvn test

# CI環境（並列実行）
mvn test -Pci
```

> **チェックポイント:**
> - 両方の環境でテストがすべてパスしますか？
> - 修正前と修正後のテストコードの違いは何ですか？

### ステップ5: 振り返り（10分）

以下の問いについて考えてみましょう。

1. **PBI実装からテスト生成への流れはどうでしたか？**
   - プロダクトコードの実装は期待通りにできましたか？
   - テスト生成時、実装したコードに対して適切なテストが生成されましたか？
   - AIにテストを依頼するとき、プロンプトの書き方がテストの品質にどう影響しましたか？

2. **なぜAIは「効率的だが壊れやすい」テストを生成したのですか？**
   - プロンプトのどの部分がそれを誘導しましたか？
   - 「テストの効率化」と「テストの信頼性」のトレードオフは？

3. **なぜローカルではテストが通るのにCI環境では失敗したのですか？**
   - ローカルとCI環境のテスト実行の違いは？
   - テストの実行順序に依存したコードはなぜ危険ですか？

4. **テストケースの「独立性」とは何ですか？（F.I.R.S.T原則のI: Independent）**
   - 各テストメソッドは他のテストの結果に依存すべきですか？
   - `static`フィールドでテスト間のデータを共有するとどんな問題が起きますか？
   - `@BeforeEach`で毎回データを初期化することの重要性は？

5. **AI駆動開発において、生成されたテストの品質をどう評価しますか？**
   - 「ローカルで通る」は十分な品質基準ですか？
   - AIが生成したコードを鵜呑みにするリスクは？
   - テストの独立性は、AIに任せられますか？開発者が担保すべきですか？
