## PBI

### 貸出上限の変更

**ユーザーストーリー:**

> 図書館の運営者として、会員の貸出上限を引き上げたい。そうすることで、利用者の満足度を向上できる。

**Acceptance Criteria:**

| # | 条件 | 変更前 | 変更後 |
|---|------|--------|--------|
| AC-1 | 一般会員の貸出上限 | 3冊 | 5冊 |
| AC-2 | プレミアム会員の貸出上限 | 10冊 | 15冊 |

> **注意:** 貸出期間（一般14日、プレミアム30日）は変更しません。

---

## 手順

### ステップ1: PBI実装（10分）

#### 1-1. プロダクトコードの変更

AIに以下の指示を出して、貸出上限を変更してください。

> **プロンプト例:**
> ```
> 以下のPBIに基づいて、貸出上限を変更してください。
> 
> 変更内容:
> - 一般会員の貸出上限: 3冊 → 5冊
> - プレミアム会員の貸出上限: 10冊 → 15冊
> 
> LendingPolicy.javaの定数を変更してください。
> テストコードはまだ変更しないでください。
> ```

#### 1-2. ユニットテストの実行

変更後、ユニットテストを実行してください。

```bash
mvn test
```

> **確認ポイント:**
> - テストは通りましたか？失敗しましたか？
> - 失敗したテストのエラーメッセージを読んでください
> - エラーメッセージから、何を修正すべきか分かりますか？

> **プロンプト例:**

> - プレミアム会員の貸出上限: 15冊

#### 2-2. 修正後の確認


> **確認ポイント:**

### ステップ3: 全テスト実行（E2E含む）（15分）

#### 3-1. E2Eテストを含む全テスト実行

ユニットテストが通ったので安心ですね。では、E2Eテストも含めて全テストを実行してみましょう。

```bash
mvn test -Dsurefire.excludedGroups=""
```

> **確認ポイント:**
> - テスト結果はどうでしたか？
> - 実行時間は何秒かかりましたか？
> - 失敗したテストはありますか？どのテストクラスですか？

#### 3-2. エラーの調査

E2Eテストが失敗した場合、以下を確認してください。

1. **エラーメッセージの比較**
   - ステップ1のユニットテストのエラーメッセージ: どんなメッセージでしたか？
   - 今回のE2Eテストのエラーメッセージ: どんなメッセージですか？
   - どちらのメッセージから原因が特定しやすいですか？

2. **E2Eテストの失敗数**
   - 何個のE2Eテストが壊れましたか？
   - ユニットテストは何個壊れましたか？
   - なぜE2Eテストの方が多く壊れたのですか？

> **ヒント:**
> ユニットテストのエラー: `expected: <3> but was: <5>` → 原因が一目瞭然
> E2Eテストのエラー: `Expected status code <409> but was <201>` → 何が起きた？

#### 3-3. E2Eテストの修正

E2Eテストを修正してください。

> **問いかけ:**
> - ユニットテストの修正と比べて、E2Eテストの修正はどうでしたか？
> - 修正にかかった時間は？
> - デバッグのしやすさは？

### ステップ5: 改善と振り返り（15分）

#### 5-1. テストの改善

AIに以下の指示を出して、テスト構成を改善してください。

> **プロンプト例:**
> ```
> LendingPolicyE2ETest.javaを分析してください。
> 
> このテストはREST API経由のE2Eテストですが、検証している内容の大半は
> ドメインロジック（LendingPolicyクラス）の単体テストで検証できます。
> 
> 以下を実施してください:
> 1. LendingPolicyE2ETestで検証している境界値・エッジケースのうち、
>    ドメインロジックで検証可能なものを、LendingPolicyTestに追加してください
> 2. LendingPolicyE2ETestにはREST APIの結合確認として最低限必要なテストだけ残してください
>    （例: 貸出APIの正常系、上限超過時のエラーステータス、存在しないリソースのエラー）
> ```

#### 5-2. 振り返り

以下の問いについて考えてみましょう。

1. **今回の体験を振り返って**
   - プロダクトコードを1行変更しただけで、何が起きましたか？
   - ユニットテストとE2Eテスト、修正のしやすさに違いはありましたか？
   - 同じロジックを2箇所でテストすることのコストは？

2. **テストピラミッドの原則**
   - ユニットテスト: ビジネスルール、境界値、計算ロジック（高速、明確なエラー）
   - 統合テスト: サービス間の連携、データベース操作
   - E2Eテスト: APIの結合確認、ステータスコード（少数に絞る）

3. **テスト層の選択基準**
   - 「このロジックは、メソッドを直接呼んで検証できるか？」→ Yes ならユニットテスト
   - 「HTTP通信を通さないと検証できないか？」→ Yes ならE2Eテスト
   - 境界値テストはどの層で書くべきですか？

4. **AI駆動開発での注意点**
   - AIに「テストを書いて」と依頼するとき、テストの層を指定していますか？
   - テストピラミッドを意識した指示とは？
