# 課題1 Variant A: 会員種別変更エッジケース - 講師ガイド

## ゴール

カバレッジが高くてもバグを見逃すテストの問題に気づき、バグ注入を通じてテスト品質を改善する方法を学ぶ。  
**Variant A固有の学び:** PBIに明記された要件（AC-8: 貸出時の会員種別で計算）であっても、AIが正しく実装・テストするとは限らないことを体験する。

## Variant Aの特徴

### PBIの罠

Variant AのPBIにはAC-8が含まれている:

> AC-8: 延滞料金は**貸出時点の会員種別**に基づいて計算する（返却時に会員種別が変更されていても、貸出時の種別を適用する）

**想定される展開:**
- AIはAC-8を読んでも、**現在の会員種別**で計算するコードを生成する可能性が高い
- なぜなら、既存の `LoanService.returnBook()` は `memberRepository` から会員を取得し、その時点の `memberType` を使うため
- AC-8を正しく実装するには、Loanモデルに `memberType` を保存する設計変更が必要だが、AIがそこまで踏み込むかは不確定
- テストも「直接MemberTypeを渡す」形式になるため、この統合レベルのバグを検知できない

### ステップ1.6で発見されるバグ

| 手順 | 操作 | 期待 |
|------|------|------|
| プレミアム会員で貸出 | `memberId:2, bookId:2` | dueDate = 30日後 |
| due_dateを10日前に変更 | SQL UPDATE | 10日延滞状態を再現 |
| 会員種別をGENERALに変更 | PUT /api/members/2 | プレミアム期限切れを想定 |
| 返却 | POST /api/loans/2/return | **期待: 200円**（プレミアム10日×20円） |
| | | **実際: 500円**（一般10日×50円） |

**根本原因:** `LoanService.returnBook()` が返却時点の会員種別を使って計算している。貸出時の会員種別がLoanに保存されていないため、貸出後に会員種別が変わると誤った料金になる。

### 想定されるAIの実装パターン

**パターン1（最も多い）: 返却時の会員種別を使用**
```java
// LoanService.returnBook() 内
Member member = memberRepository.findById(loan.getMemberId())...;
int fee = OverdueFeePolicy.calculateOverdueFee(
    member.getMemberType(),  // ← 返却時の種別を使用（バグ）
    loan.getDueDate(), loan.getReturnDate());
```

**パターン2（稀）: Loanに貸出時の会員種別を保存**
```java
// Loan に memberType フィールドを追加
// borrowBook() で loan.setMemberType(member.getMemberType())
// returnBook() で loan.getMemberType() を使用
```
→ パターン2の場合、AC-8が正しく実装される。この場合はステップ1.6でバグは出ないが、テストの品質問題は依然として存在する。

## よくあるAI生成テストの問題

**パターン1: 存在確認のみ**
```java
assertNotNull(fee);
assertTrue(fee.amount() > 0);  // 50円でも999円でもパス
```

**パターン2: 範囲確認のみ**
```java
assertTrue(fee.amount() <= 1000);  // 800円でも500円でもパス
```

**パターン3: 例外の型のみ**
```java
assertThrows(IllegalArgumentException.class, ...);  // メッセージ未検証
```

**改善後（目指す形）:**
```java
assertEquals(250, fee.amount());  // PBIの計算例を具体的に検証
assertEquals(1000, fee.amount());  // 上限値を具体的に検証
assertTrue(exception.getMessage().contains("返却日"));  // メッセージも検証
```

## キーメッセージ

**仕様とテストのギャップ**
- PBIに書かれた要件（AC-8）であっても、AIが正しく実装・テストするとは限らない
- 開発者がPBIの要件を理解し、テストでカバーされているか確認する責任がある

**カバレッジ vs 品質**
- カバレッジが高い ≠ テストが機能している
- AIは「行を実行するテスト」を書けるが、「ビジネスルールを検証するテスト」を書くとは限らない

**バグ注入テスト（ミューテーションテスト）の価値**
- テストの品質を可視化する強力な手法
- 「存在確認」→「値の検証」で検知率が大幅に向上する

**AI時代の開発者スキル**
- AIが生成したコードとテストの品質を評価・改善できるのは開発者だけ
- PBIの要件をテストに落とし込む能力がより重要に
- 「AIが書いたから大丈夫」ではなく、「仕様通りか自分で確認する」姿勢が必要
