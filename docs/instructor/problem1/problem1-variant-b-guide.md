## Variant Bの特徴

### PBIの罠

Variant BのPBIには、以下が**意図的に欠落**している:

- **AC-5（期限内返却で0円）が存在しない** — 期限前返却の扱いが未定義
- **計算例に期限前返却がない** — すべての計算例が延滞ケースのみ
- **AC-5の延滞日数定義が単純** — `延滞日数 = 返却日 - 貸出期限日` とだけ記載

**想定される展開:**
- AIはPBIに忠実に `returnDate - dueDate` を計算する
- 期限前返却の場合、この値がマイナスになる
- `Math.min(マイナス値, 上限)` はマイナス値をそのまま返す
- 結果として延滞料金がマイナス（例: -700円）になる

### ステップ1.6で発見されるバグ

| 手順 | 操作 | 期待 |
|------|------|------|
| 会員登録 | 一般会員を1名登録 | 会員ID取得 |
| 書籍登録 | 書籍を1冊登録 | 書籍ID取得 |
| 貸出 | 書籍を貸出 | dueDate = 14日後 |
| **すぐ返却** | 同日に返却 | **期待: 0円** |
| | | **実際: -700円**（-14日 × 50円） |

**根本原因:** `OverdueFeePolicy.calculateOverdueFee()` が延滞日数のマイナス値をチェックしていない。PBIに「期限前返却は0円」という仕様がなく、AIはPBIに書かれたことだけを忠実に実装した。

### 想定されるAIの実装

**典型的な実装（バグあり）:**
```java
public static int calculateOverdueFee(MemberType memberType, LocalDate dueDate, LocalDate returnDate) {
    long overdueDays = ChronoUnit.DAYS.between(dueDate, returnDate);  // マイナスになりうる
    int fee = (int) overdueDays * getDailyFee(memberType);
    return Math.min(fee, getMaxFee(memberType));  // Math.min(-700, 1000) = -700
}
```

**正しい実装（バグ修正後）:**
```java
public static int calculateOverdueFee(MemberType memberType, LocalDate dueDate, LocalDate returnDate) {
    long overdueDays = ChronoUnit.DAYS.between(dueDate, returnDate);
    if (overdueDays <= 0) {
        return 0;
    }
    int fee = (int) overdueDays * getDailyFee(memberType);
    return Math.min(fee, getMaxFee(memberType));
}
```

### AIが自主的にバグを防いだ場合

AIが `Math.max(0, overdueDays)` や `if (overdueDays <= 0) return 0;` を自主的に追加することがある。その場合:

1. 返却結果は正しく0円になる
2. **しかし、テストコードを確認させる** — 「期限前返却のテスト」は書かれているか？
3. テストがなければ「たまたま正しいだけ」で、リファクタリング時に壊れる可能性がある
4. ステップ2のミューテーションテストで、テスト品質の問題は同様に露呈する

## よくあるAI生成テストの問題

**パターン1: 存在確認のみ**
```java
assertNotNull(fee);
assertTrue(fee.amount() > 0);  // 50円でも999円でもパス
```

**パターン2: 範囲確認のみ**
```java
assertTrue(fee.amount() <= 1000);  // 800円でも500円でもパス
```

**パターン3: 例外の型のみ**
```java
assertThrows(IllegalArgumentException.class, ...);  // メッセージ未検証
```

**パターン4: 延滞ケースだけテスト**
```java
// すべてのテストが「延滞日数 > 0」のケースだけ
// 期限前返却（延滞日数 < 0）のテストが存在しない
```

**改善後（目指す形）:**
```java
assertEquals(250, fee.amount());  // PBIの計算例を具体的に検証
assertEquals(0, fee.amount());    // 期限前返却は0円
assertEquals(0, fee.amount());    // 期限当日も0円
assertTrue(exception.getMessage().contains("返却日"));  // メッセージも検証
```

### トラブルシューティング

## キーメッセージ

**仕様の空白を埋める責任**
- PBIに「書かれていないこと」を発見するのは開発者の仕事
- AIはPBIに忠実だが、仕様の空白を指摘してくれるとは限らない
- 「期限前返却はどうなるか？」という問いを立てられるかが開発者の腕の見せどころ

**カバレッジ vs 品質**
- カバレッジ100% ≠ テストが機能している
- AIは「行を実行するテスト」を書けるが、「ビジネスルールを検証するテスト」を書くとは限らない
- PBIの計算例がすべて延滞ケースなら、テストも延滞ケースだけになる

**ミューテーションテストの価値**
- テストの品質を可視化する強力な手法
- 「存在確認」→「値の検証」で検知率30% → 90%

**AI時代の開発者スキル**
- AIが生成したコードとテストの品質を評価・改善できるのは開発者だけ
- 仕様に書かれていないエッジケースを想像する力がより重要に
- 「テストが通った = 正しい」ではなく、「何をテストしていないか」を考える姿勢が必要
