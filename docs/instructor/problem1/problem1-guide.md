# 課題1: 延滞料金計算機能の追加 - 講師ガイド

## ゴール

AIが生成したテストがパスしカバレッジも高い状態でも、手動テストでバグを発見できることを体験し、AI生成テストの限界と開発者によるレビューの重要性を学ぶ。

## 構成（45分）

| ステップ | 内容 | 時間 |
|---------|------|------|
| 1 | 実装コードの確認とテスト作成 | 20分 |
| 1.5 | エッジケースの動作確認 | 10分 |
| 2 | バグの修正とテストの改善 | 10分 |
| 3 | 振り返り | 5分 |

## 想定される展開

1. **テスト全パス＋高カバレッジ達成** → 安心感
2. **手動テストでバグ発見**（期限前返却なのに延滞料金が発生） → 衝撃
3. **バグ修正＋テスト追加** → AI生成テストだけでは不十分という気づき

## 各ステップの講師ポイント

### ステップ1: 実装コードの確認とテスト作成（20分）

- 実装コードは既に用意済み。受講者にはまずコードを読んでもらう
- AIにテストを生成させる際、PBIのAcceptance Criteriaと計算例をプロンプトに含めるよう誘導する
  - 仕様は延滞料金の計算について書かれており、期限前に返却した際の仕様は明言されてない。ただ常識的には0円になるので、明示されていなくても参加者は違和感を感じない。
- テスト全パス＋カバレッジ確認まで進めてもらう

**問いかけ:** 「テストが全部パスしましたね。カバレッジも高いです。これで安心ですか？」

### ステップ1.5: エッジケースの動作確認（10分）

- フロントエンドからの手動テストで、**貸出直後に返却**してもらう
- 期限前に返却した際、延滞料金が負の値になるバグに気づかせる

**問いかけ:** 「返却期限はまだ先のはずですが、延滞料金はどうなっていますか？」

### ステップ2: バグの修正とテストの改善（10分）

- 発見したバグをAIに修正させる
- このバグを検知できるテストケースも追加させる

**問いかけ:** 「なぜAI生成テストはこのバグを見逃したのでしょうか？」

### ステップ3: 振り返り（5分）

グループ議論のテーマ:

1. **AIによいテストを書かせるには？** — プロンプトの内容、エッジケースの指示、開発者が確認すべきこと
2. **カバレッジとテストの品質** — カバレッジが高い ≠ テストが信頼できる、コード実行と正しさの検証の違い

## よくあるAI生成テストの問題

| パターン | 例 | 問題点 |
|---------|---|--------|
| 存在確認のみ | `assertNotNull(fee)` | どんな値でもパスする |
| 範囲確認のみ | `assertTrue(fee >= 0)` | 誤った金額でもパスする |
| 正常系のみ | 延滞ありケースだけテスト | 期限前返却のケースが漏れる |

## トラブルシューティング

| 問題 | 対応 |
|------|------|
| ポートが表示されない | VS Code下部「ポート」タブから38080を確認 |
| カバレッジが更新されない | `mvn test jacoco:report` 再実行、ブラウザキャッシュクリア |

## キーメッセージ

- **カバレッジ100% ≠ バグがない。** AIは行を実行するテストは書けるが、ビジネスルールの検証を保証しない
- **手動テスト・レビューは依然重要。** AI生成テストの品質を評価・改善できるのは開発者だけ
- **プロンプトの質がテストの質を左右する。** Acceptance Criteriaや計算例を含めることで精度が上がる
