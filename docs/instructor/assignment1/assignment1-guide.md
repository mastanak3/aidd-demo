# 課題1: 延滞料金計算機能のテスト - 講師ガイド

## ゴール

AIが生成したテストがパスしカバレッジも高い状態でも、開発者が「観点」を与えることで初めてバグを検出できることを体験し、AI生成テストの限界と開発者によるレビューの重要性を学ぶ。

## 構成（60分）

| ステップ | 内容 | 時間 |
|---------|------|------|
| 1 | 実装コードの確認とテスト作成 | 20分 |
| 2 | テストの観点を深掘りする | 20分 |
| 3 | バグの修正とテストの改善 | 15分 |
| 4 | 振り返り | 5分 |

## 想定される展開

1. **テスト全パス＋高カバレッジ達成** → 安心感
2. **開発者が観点を与えてテスト追加** → テストが失敗し始める → 衝撃
3. **バグ修正＋テスト改善** → 「AIに何を伝えるか」が品質を決めるという気づき

## 埋め込まれたバグの一覧

`LendingPolicy.java` の `calculateOverdueFee` メソッドには以下の3つのバグが埋め込まれている。

### バグ1: `Period.between().getDays()` による月跨ぎの計算ミス

- **該当箇所:** `LendingPolicy.java` 43行目
- **コード:** `long calendarDays = Period.between(dueDate, returnDate).getDays();`
- **問題:** `Period.between()` は年・月・日の3成分を持つ `Period` を返し、`.getDays()` は日の成分だけを返す。月を跨ぐと月の成分に吸収された日数が無視される
- **例:** 期限日5/7 → 返却日6/11 の場合、`Period` = `P1M4D` → `.getDays()` = **4**（正しくは35日）
- **修正:** `ChronoUnit.DAYS.between(dueDate, returnDate)` を使う
- **検出する観点:** 観点A（延滞期間が月を跨ぐ場合）

### バグ2: `countClosedDays` の走査範囲ずれ

- **該当箇所:** `LendingPolicy.java` 62〜71行目
- **コード:** `from`（= dueDate）をinclusive、`to`（= returnDate）をexclusiveで走査
- **問題:** 延滞日数の対象期間は「期限日の翌日〜返却日」だが、コードは「期限日〜返却日の前日」を走査している。1日ずれている
- **影響:**

  | ケース | 影響 |
  |--------|------|
  | 期限日が月曜 | 期限日自体を休館日カウント → 延滞日数が過少 |
  | 返却日が月曜 | 返却日がカウント対象外 → 延滞日数が過大 |

- **例1（期限日が月曜）:** 一般会員、期限日=5/12(月), 返却日=5/14(水)

  | | calendarDays | closedDays | overdueDays | 延滞料金 |
  |---|---|---|---|---|
  | バグあり | 2 | 1（5/12を誤カウント） | **1日** | 50円 |
  | 正しい計算 | 2 | 0（延滞期間5/13,5/14に月曜なし） | **2日** | 100円 |

- **例2（返却日が月曜）:** 一般会員、期限日=5/7(水), 返却日=5/12(月)

  | | calendarDays | closedDays | overdueDays | 延滞料金 |
  |---|---|---|---|---|
  | バグあり | 5 | 0（5/12が走査対象外） | **5日** | 250円 |
  | 正しい計算 | 5 | 1（5/12は月曜） | **4日** | 200円 |
- **修正:** 走査範囲を `dueDate.plusDays(1)` 〜 `returnDate`（inclusive）に修正
- **検出する観点:** 観点B（期限日や返却日が休館日の場合）

### バグ3: ブックポスト返却 × 休館日の複合バグ

- **該当箇所:** `LoanService.java` の返却処理と `LendingPolicy.java` の相互作用
- **問題:** 火曜日の午前9時前にブックポスト返却すると、返却日が前日の月曜日（休館日）になる。バグ2の走査範囲ずれと組み合わさり、延滞料金の計算がさらに不正確になる
- **性質:** AC-7（休館日の除外）× AC-8（ブックポスト返却）の交差点で発生する複合バグ。個別のACは正しく実装されているように見えるが、組み合わせで問題が生まれる
- **例:** 一般会員、期限日5/7(水)、火曜5/13の午前9時前にブックポスト返却 → 返却日は5/12(月)

  | | calendarDays | closedDays | overdueDays | 延滞料金 |
  |---|---|---|---|---|
  | バグあり | 5 | 0（5/12が走査対象外） | **5日** | 250円 |
  | 正しい計算 | 5 | 1（5/12は月曜） | **4日** | 200円 |

  バグ2の走査範囲 `[dueDate, returnDate)` により、returnDateである5/12(月)が休館日としてカウントされず、延滞が1日多くなる
- **検出する観点:** 観点C（ブックポスト返却で前日が休館日になるケース）

## 各ステップの講師ポイント

### ステップ1: 実装コードの確認とテスト作成（20分）

- 実装コードは既に用意済み。受講者にはまずコードを読んでもらう
- AIにテストを生成させる際、PBIのAcceptance Criteriaと計算例をプロンプトに含めるよう誘導する
- 計算例はすべて**同じ月内・期限日/返却日が月曜以外**になっている点に注目（これが後の伏線）
- テスト全パス＋カバレッジ確認まで進めてもらう
- この時点でAIが生成するテストは、仕様の計算例をなぞるだけのものになりがち。カバレッジは高いがバグは検出できない

**問いかけ:** 「テストが全部パスしましたね。カバレッジも高いです。これで安心ですか？」

### ステップ2: テストの観点を深掘りする（20分）

ここが本課題の核心。開発者が「観点」をAIに伝えることで、仕様の計算例だけでは見つからないバグを検出する。

#### 観点A: 延滞期間が月を跨ぐ場合（→ バグ1を検出）

- 仕様の計算例はすべて同じ月内で完結している点に気づかせる
- 月跨ぎのテストを追加すると、延滞日数が極端に少なくなりテストが失敗する
- 原因は `Period.between().getDays()` が日の成分しか返さないこと
- Java の日付APIに詳しくなくても「期待値と実際の値の差」からバグに気づける

**問いかけ:** 「計算例はすべて5月内でしたね。月を跨いだらどうなるでしょう？」

#### 観点B: 期限日や返却日が休館日（月曜日）の場合（→ バグ2を検出）

- 仕様の計算例では期限日・返却日がすべて月曜以外であることに気づかせる
- 期限日が月曜のテストを追加すると、休館日カウントがずれてテストが失敗する
- `countClosedDays` の走査範囲が `[dueDate, returnDate)` になっていて、正しくは `(dueDate, returnDate]` であるべき

**問いかけ:** 「計算例の曜日を見てください。期限日や返却日が月曜のケースはありますか？」

#### 観点C: ブックポスト返却で前日が休館日になるケース（→ バグ3を検出）

- AC-7とAC-8の組み合わせで生まれる複合バグ
- 火曜の午前9時前に返却 → 返却日が月曜（休館日）→ バグ2と相互作用
- 「仕様の交差点」にあるバグの典型例

**問いかけ:** 「ブックポスト返却で前日が月曜になるケース、考えましたか？」

### ステップ3: バグの修正とテストの改善（15分）

- ステップ2で発見したバグをAIに修正させる
- バグ1の修正は `ChronoUnit.DAYS.between()` への置き換えで比較的単純
- バグ2の修正は走査範囲の変更。修正後に既存テストが壊れないことも確認させる
- バグ3はバグ2の修正で連動して直る可能性があるが、テストケースとしては残すべき

**問いかけ:** 「なぜステップ1のAI生成テストはこれらのバグを見逃したのでしょうか？」

### ステップ4: 振り返り（5分）

グループ議論のテーマ:

1. **AIに渡す「観点」の重要性**
   - ステップ1のテストは仕様の計算例をなぞっただけ。計算例にない境界条件はテストされない
   - 開発者が「月跨ぎ」「境界日が休館日」「ACの組み合わせ」といった観点を与えて初めて検出できた
2. **カバレッジとテストの品質**
   - カバレッジが高い ≠ バグがない。コードを実行するテストと正しさを検証するテストは別物
3. **仕様の「交差点」にあるバグ**
   - 個別のACは正しくても、ACの組み合わせ（AC-7 × AC-8）でバグが生まれる
   - このような交差点は仕様書に明示されないことが多く、開発者が能動的に特定する必要がある

## よくあるAI生成テストの問題

| パターン | 例 | 問題点 |
|---------|---|--------|
| 仕様の計算例の転記 | 計算例の4パターンだけテスト | 計算例にないケースが漏れる |
| 存在確認のみ | `assertNotNull(fee)` | どんな値でもパスする |
| 範囲確認のみ | `assertTrue(fee >= 0)` | 誤った金額でもパスする |
| 同一月内のみ | 5月内のテストだけ | 月跨ぎの `Period.getDays()` バグを見逃す |
| 非休館日のみ | 期限日・返却日が火〜日 | 月曜境界の走査範囲バグを見逃す |

## トラブルシューティング

| 問題 | 対応 |
|------|------|
| ポートが表示されない | VS Code下部「ポート」タブから38080を確認 |
| カバレッジが更新されない | `mvn test jacoco:report` 再実行、ブラウザキャッシュクリア |
| 月跨ぎテストで期待値がわからない | カレンダーで月曜を数えて手計算させる。計算過程を確認することも学び |
| バグ修正で既存テストが壊れる | 走査範囲の修正で計算例の結果が変わらないことを確認させる |

## キーメッセージ

- **カバレッジ100% ≠ バグがない。** AIは行を実行するテストは書けるが、ビジネスルールの検証を保証しない
- **「観点」を与えるのは開発者の仕事。** AIは指示された範囲では高品質なテストを書けるが、テスト観点の網羅性は開発者が担保する
- **仕様の「交差点」を意識する。** 個別のACが正しくても、組み合わせでバグが生まれる。この交差点を見つけるのは開発者の役割
- **プロンプトの質がテストの質を左右する。** Acceptance Criteria・計算例に加え、境界条件やACの組み合わせを明示的に伝えることで精度が上がる
