## ワークショップフロー

### 題材: 図書貸出管理システム

シンプルなCRUDでありながら、会員種別ごとの貸出上限・延滞料金・貸出期間など細かいドメインルールが自然に生まれる題材です。

---

### フェーズ0（事前準備・提供済み）: 適切に設計されたベースライン

参加者に提供する初期プロジェクト。正しい設計のお手本として機能する。

**機能:**
- 書籍CRUD（タイトル、著者、ISBN、貸出状態）
- 会員CRUD（氏名、メールアドレス、会員種別: 一般/プレミアム）
- 貸出・返却（書籍を借りる・返す）

**ドメインルール:**
- 一般会員: 同時貸出上限3冊、貸出期間14日
- プレミアム会員: 同時貸出上限10冊、貸出期間30日

**設計ポイント:**
- ドメインロジックはPOJO（`LendingPolicy`、`Member`、`Book`等）に隔離
- Resource → ApplicationService → Domain の層構造
- テストピラミッド: ドメインの単体テスト、Weld-Testingによる結合テスト、E2Eテスト
- CI/CDパイプライン（Gitea Actions等）が稼働済み
- SonarQubeによる静的解析がCIに組み込まれており、コード品質（コードスメル、複雑度、カバレッジ、重複等）が可視化される

---

### 導入（15分）

- ワークショップの目的と全体の流れを説明
- フェーズ0のコードウォークスルー（設計意図、テスト構成、CI/CD・SonarQubeのダッシュボードを確認）
- Claude Codeの基本操作を確認（全フェーズでAIを使って実装する旨を伝える）
- 「なぜこの設計になっているか」を問いかけ、後のフェーズで実感してもらう布石を打つ

---

### フェーズ1（50分）: 新機能PBIの実装 ― AIにおまかせで罠にはまる

**PBI: 「延滞料金の計算機能を追加する」**

| 項目   | 内容                                                     |
| ------ | -------------------------------------------------------- |
| 要件   | 返却時、貸出期間を超過していれば延滞料金を計算する       |
| ルール | 一般会員: 1日あたり100円、プレミアム会員: 1日あたり200円 |
| 記録   | 延滞料金を返却履歴に記録する                             |

**進め方:** PBIの内容をClaude Codeに渡して実装させる。CLAUDE.mdの整備や設計方針の指示は**あえて行わない**。

**AIが取りがちな"悪手"（意図的に放置する）:**
- `LendingService`に延滞料金計算ロジックを直接書く（ドメイン分離しない）
- テストで`LendingService`の内部メソッド呼び出しをモックで検証する（実装の詳細に依存）
- E2Eテストを書かずにユニットテストだけで済ませる
- SonarQubeの指摘（複雑度の増加、コードスメル等）を無視して先に進む

**狙い:** AIが生成したコードは動くので問題に気づかない。設計ガイドラインなしのAIは「動くが壊れやすいコード」を量産する。フェーズ2で痛みが来る。

**講師の介入ポイント:**
- あえて設計指針やCLAUDE.mdの作成を促さず、参加者の自然なAI活用を尊重する
- 「テストは通っていますか？CI通っていますか？SonarQubeの結果は見ましたか？」だけ確認する

### フェーズ2（60分）: 要件変更 ― ガイドなしAIの痛みを体感する

**PBI: 「貸出ルールの大幅変更」**

| 変更項目     | 内容                                                         |
| ------------ | ------------------------------------------------------------ |
| 新会員種別   | 「学生会員」を追加（上限5冊、貸出期間21日、延滞料金50円/日） |
| 延滞計算変更 | 土日祝日は延滞日数に含めない                                 |
| 貸出期間     | 会員種別ごとに管理者が変更可能にする                         |

**進め方:** 引き続きClaude Codeで実装する。フェーズ1でAIが生成したコードに対して要件変更を適用させる。

**ここで発生する"痛み":**

1. **ドメイン分離していない場合:**
   - AIが`LendingService`にif分岐を乱立させ、新会員種別の追加がサービス層全体に波及
   - 延滞計算の変更が複数箇所に散らばり、AIも修正漏れを起こす

2. **実装の詳細に依存したテストを書いた場合:**
   - AIがリファクタリング（料金計算のStrategy化等）を試みるとテストが大量に壊れる
   - AIがテスト修正に時間を取られ、振る舞いは変わっていないのにテストの書き直しが発生する

3. **テストピラミッド/CI/CDが不十分な場合:**
   - AIが変更の影響範囲を正しく把握できず、意図しないデグレが発生
   - AIの出力を人間が検証するコストが増大する

4. **SonarQubeの指摘が蓄積している場合:**
   - フェーズ1から持ち越した複雑度・コードスメルが要件変更でさらに悪化
   - SonarQubeのダッシュボードで品質の劣化が数値として可視化される

**講師の介入ポイント（中間振り返り 10分）:**
- 「AIが生成したコードで、テストが壊れた人はいますか？なぜ壊れましたか？」
- 「振る舞いのテストとは何か」「ドメインをPOJOに隔離する意味」を解説
- フェーズ0のコードがなぜこの変更に耐えられる設計だったかを振り返る
- AIは既存コードの慣習を踏襲する → フェーズ1で崩れた設計がフェーズ2で増幅されたことを確認
- SonarQubeのダッシュボードでフェーズ0→1→2の品質推移を比較し、数値で劣化を確認

### フェーズ3（30分）: AI駆動開発 ― 正しいガイドで品質を取り戻す

**PBI: 「予約機能の追加」**

| 項目   | 内容                                                       |
| ------ | ---------------------------------------------------------- |
| 要件   | 貸出中の書籍に対して予約ができる。返却時、予約者に通知する |
| ルール | 同一書籍への予約は先着順、会員あたり予約上限2件            |

**デモ＋ハンズオン:**
- 講師が **きちんと設計されたコードベース＋AIへのガイド設定** に対してClaude Codeで実装するデモを見せる
  - CLAUDE.mdによる設計方針・テスト方針の言語化、カスタムスキルによるワークフローの定型化など、AIを正しく導く仕組みを示す
  - フェーズ1-2と同じ「AIに任せる」のに、出力品質が大きく異なることを可視化する
- 参加者もCLAUDE.mdやカスタムスキルを整備した上で、自分のコードベースに対してClaude Codeで実装を試みる
  - ガイドラインを与えた場合と与えなかった場合（フェーズ1-2）の差を実感する

**学びのポイント:**
- 全フェーズでAIを使ったからこそわかる: AIの出力品質を決めるのは「コードベースの設計」と「言語化されたガイドライン」
- CLAUDE.md・カスタムスキル・ADR等による設計方針やワークフローのテキスト化がAI活用の前提条件であること
- SonarQubeの指摘をAIへのフィードバックに活用できる（指摘を渡して修正させる等）
- AIは万能ではなく、人間が設計の基礎を理解し正しくガイドする必要がある

### 振り返り・まとめ（25分）

**グループ振り返り（15分）:**
- フェーズ1-2とフェーズ3で、AIの出力にどんな違いがあったか？
- AIが良いコードを書くために、人間側に何が求められるか？
- 設計の基礎（ドメイン分離、振る舞いのテスト等）がAI活用にどう影響したか？

**講師まとめ（10分）:**

| 原則             | ワークショップでの体験                                           |
| ---------------- | ---------------------------------------------------------------- |
| 振る舞いのテスト | 実装を変えてもテストが壊れない → AIのリファクタリングを安全に受け入れられる |
| ドメインの隔離   | 要件変更の影響範囲がPOJOに閉じる → AIが変更箇所を正しく特定できる |
| テストピラミッド | 高速なフィードバック → AIの出力を即座に検証できる                 |
| 静的解析の活用   | SonarQubeの指摘が品質劣化の早期発見とAIへのフィードバックに役立つ  |
| 設計方針の言語化 | CLAUDE.md・カスタムスキル等があるとAIが一貫した品質のコードを出力する |
