# AI 駆動開発ワークショップ — 設計方針

## ワークショップの目的

図書貸出管理システムへの機能追加（PBI 実装）ハンズオンを通じて、**ソフトウェアエンジニアリングの基礎 — とくに古典学派テスト — は AI の時代でも変わらず重要である**ことを体験的に学んでもらう。

AI コーディングツールがコードを高速に生成できる時代だからこそ、生成されたコードの正しさを担保するテスト設計力・品質判断力がエンジニアに求められる。本ワークショップでは、受講者が自ら手を動かしてテストを書き、壊し、直す過程でその感覚を身につけることを目指す。

---

## 課題構成

全 4 課題。各課題は独立した PBI（機能追加）を題材に、テストに関する特定のトピックを掘り下げる。

### 課題 1: テスト観点の重要性

| 項目 | 内容 |
|------|------|
| **PBI** | 延滞料金計算機能 |
| **学習目標** | エッジケースの挙動をテストすることの重要性を理解する。カバレッジが高くても、仕様上の境界値やエッジケースが漏れていればバグは見抜けない。 |
| **トピック** | 偽陰性 / カバレッジの限界 / 仕様レベルの入力空間に関するバグ |

**ポイント:** カバレッジ 100% でも安心できない。テストが「何を検証しているか」を意識しなければ、偽陰性（バグがあるのにテストが通る）を見逃す。

---

### 課題 2: テストダブルの導入判断

| 項目 | 内容 |
|------|------|
| **PBI** | ISBN 検索・補完機能 |
| **学習目標** | フレーキーなテストが開発現場にもたらす混乱を体感する。テストダブルを採用する判断基準を理解する。 |
| **トピック** | フレーキーテスト / テストダブル / 管理外依存 |

**ポイント:** 外部サービスへの依存があるテストは非決定的になりやすい。管理外依存に対してテストダブルを導入する判断と、その設計を学ぶ。

---

### 課題 3: テストの独立性と並列実行

| 項目 | 内容 |
|------|------|
| **PBI** | 貸出延長機能 |
| **学習目標** | テストの Setup/Teardown を適切に行い独立性を保証することの重要性を理解する。ローカルと CI 環境でテストの実行方法が変わる可能性があることを知る。 |
| **トピック** | Setup/Teardown / テストの独立性 |

**ポイント:** テスト間で状態を共有すると、実行順序やタイミングによって結果が変わる。逐次実行では気づかなかった問題が並列実行で顕在化するケースを体験する。

---

### 課題 4: テストピラミッドと高速フィードバック

| 項目 | 内容 |
|------|------|
| **PBI** | 貸出上限の変更 |
| **学習目標** | テストのスコープ（単体・結合・E2E）とその比率を意識した設計の重要性を理解する。高速なフィードバックループがなぜ大切かを体感する。 |
| **トピック** | テストピラミッド / フィードバックループの速度 |

**ポイント:** E2E テストばかりに頼ると実行時間が膨らみ、開発のリズムが崩れる。変更に対して素早くフィードバックを得られるテスト構成を考える。

---

## 全体を通じて伝えたいこと

1. **AI が書いたコードもバグる** — 生成されたコードを鵜呑みにせず、テストで検証する習慣が不可欠。
2. **テストの「量」より「質」** — カバレッジの数値だけでなく、何をテストしているかが重要。
3. **テストもまた設計である** — テストの独立性、実行速度、信頼性は意識的に設計しなければ手に入らない。
4. **古典学派テストの原則は普遍的** — ツールや技術が変わっても、テスト設計の基本原則は変わらない。
