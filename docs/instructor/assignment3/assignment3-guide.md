# 課題3: 外部サービス連携とISBN自動検索 - 講師ガイド

## ゴール

外部サービスに依存するテストの不安定さ（Flaky Test）を体験し、「管理下にない依存」をテストダブルに置き換える原則（FIRST原則のR: Repeatable）を学ぶ。

## 想定される展開

1. **実装完了** → 既存テストパス → 安心感
2. **外部サービスを呼ぶテスト生成** → パスしたりしなかったり → 違和感
3. **テストダブルで解決** → 決定的なテストの価値を実感 → 学び

## 各ステップの講師ポイント

### ステップ1: 機能の実装（15分）

- AIがインターフェースを導入せず直接HTTP呼び出しを書く場合がある。ステップ4で気づかせればよいので、ここでは許容する
- AIがMockitoの`@Mock`を使った場合は「このプロジェクトではモックライブラリを使いません」と補足

### ステップ2: テストの作成（10分）

- 外部サービスを実際に呼ぶテストをAIに生成させる。`isbn-service-booklist.md`を参照するよう指示させる
- この時点ではテストコードとして妥当に見える。問題はステップ3で顕在化する

### ステップ3: テストの実行（10分）

外部サービスが安定してテストが常にパスしてしまう場合：
- 「外部サービスが落ちたらどうなりますか？」と問いかけて思考実験させる
- 「このテストはCI環境でも毎回同じ結果になりますか？」と問いかける

### ステップ4: 原因分析と対策（20分）

ここが本課題の核心。段階的に問いかける：

1. 「このテストの成否を決めているのは、テストコードですか？外部の何かですか？」
2. 「`IsbnLookupService`はインターフェースですね。テスト専用の実装は作れませんか？」
3. 「成功を返す実装と、必ず例外を投げる実装があれば、両方のケースをテストできませんか？」

**想定される解法:** テスト用フェイク実装（Stub）を作り、WeldInitiatorで差し替える。成功ケースと失敗ケースでテストクラスを分ける。

> LoanServiceTestのWeldInitiatorにも`ExternalIsbnLookupService`が含まれている場合、同様にStubに差し替えが必要。

## トラブルシューティング

| 問題 | 対応 |
|------|------|
| AIがインターフェースを作らず直接HTTP呼び出し | 「テスト用の偽物に差し替えるには？」と問いかけ、インターフェース導入に誘導 |
| AIがMockitoを使う | 「インターフェースの実装クラスを作ってください」と指示 |
| WeldInitiatorでのDI差し替えが失敗 | フェイクに`@ApplicationScoped`が必要。WeldInitiatorに含めるクラスの過不足を確認 |

## キーメッセージ

- **テストの決定性は開発者が設計する。** AIは非決定的な要素を自律的に排除しない
- **管理下にない依存はテストダブルに置き換える。** 管理下にある依存（DB等）は本物を使う
- **インターフェースがテスタビリティを生む。** 設計がテストの品質に直結する
