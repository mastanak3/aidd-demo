# 課題5: テストピラミッドの崩壊と改善 - 講師ガイド

## ゴール

PBI実装時にE2Eテストが大量に壊れる体験を通じて、テストピラミッドの原則（境界値はユニットテスト、E2Eは結合確認のみ）を学ぶ。

## 構成（60分）

1. **ステップ1**: PBI実装（10分）- 貸出上限の定数変更
2. **ステップ2**: ユニットテスト修正（10分）- 明確なエラー → 簡単に修正 → 安心感
3. **ステップ3**: 全テスト実行（15分）- E2Eテスト破壊 + 遅い + 不明瞭なエラー → 痛み
4. **ステップ4**: 分析（10分）- 同じロジックの二重テスト問題
5. **ステップ5**: 改善と振り返り（15分）

## 想定される展開

- **ステップ2完了**: ユニットテスト修正 → 「簡単だった、全テスト通った」→ 安心感
- **ステップ3実行**: E2Eテスト破壊 → 「えっ、まだ壊れてるの？しかも遅い...」→ 衝撃
- **エラーメッセージ比較**: ユニット `expected 3 but was 5` vs E2E `Expected 409 but was 201` → 気づき
- **ステップ5完了**: テスト移行 → 「境界値はユニットで書くべき」→ 学び

## 仕込まれた問題

### LendingPolicyE2ETest（23個のE2Eテスト）

`backend/src/test/java/com/example/library/resource/LendingPolicyE2ETest.java`

本来ユニットテストで検証すべき境界値テスト（貸出上限の0〜上限超過まで1冊ずつ）が、
すべてREST API経由のE2Eテストとして実装されている。

### PBI実装で壊れるテスト

**ユニットテスト（LendingPolicyTest）** - 4個:

| テスト | エラーメッセージ | 修正の容易さ |
|-------|----------------|------------|
| `一般会員の貸出上限は3冊` | `expected: <3> but was: <5>` | 期待値を5に変更 |
| `一般会員は貸出3冊なら追加で借りられない` | `expected: <false> but was: <true>` | 境界値を5に変更 |
| `プレミアム会員の貸出上限は10冊` | `expected: <10> but was: <15>` | 期待値を15に変更 |
| `プレミアム会員は貸出10冊なら追加で借りられない` | `expected: <false> but was: <true>` | 境界値を15に変更 |

→ エラーメッセージから原因が一目瞭然。修正は期待値の変更のみ。

**E2Eテスト（LendingPolicyE2ETest）** - 6個以上:

| テスト | エラーメッセージ | 修正の難しさ |
|-------|----------------|-------------|
| `一般会員_3冊借りている状態で4冊目は貸出不可` | `Expected status code <409> but was <201>` | テスト全体を書き換え必要 |
| `一般会員_3冊借りて1冊返却後に再度貸出できる` | テストの前提が変わり不整合 | ロジックの見直し必要 |
| `一般会員が連続で上限まで借りて超過を確認` | `Expected status code <409> but was <201>` | ループ数と期待値の変更 |
| `プレミアム会員_10冊借りている状態で11冊目は貸出不可` | `Expected status code <409> but was <201>` | 10→15冊分のデータ準備に変更 |
| `プレミアム会員_10冊借りて1冊返却後に再度貸出できる` | テストの前提が変わり不整合 | ロジックの見直し必要 |
| `プレミアム会員が連続で上限まで借りて超過を確認` | `Expected status code <409> but was <201>` | ループ数変更 + データ準備増 |

→ エラーメッセージから原因が分かりにくい。テストデータの準備やAPI呼び出し回数の変更が必要。

### 教訓のポイント

| 観点 | ユニットテスト | E2Eテスト |
|-----|-------------|---------|
| 壊れた数 | 4個 | 6個以上 |
| エラーの明確さ | `expected 3 but was 5` | `Expected 409 but was 201` |
| 修正コスト | 期待値の変更のみ | データ準備・ループ数・API呼び出し回数の変更 |
| 実行時間 | 0.03秒 | 19秒 |
| フィードバック速度 | 即座 | 19秒待ってから |

## Maven設定

E2Eテストは`@Tag("e2e")`でタグ付け:
- `mvn test` → E2Eテスト除外（ステップ2で使用）
- `mvn test -Dsurefire.excludedGroups=""` → 全テスト実行（ステップ3で使用）
- `mvn test -Pci` → 全テスト実行（CI環境）

## 正しい改善方法

### 改善前の構成

```
E2E: 23個（境界値 + 状態遷移 + エラー + 回復）
ユニット: 11個（基本テスト）
→ 逆ピラミッド型、仕様変更時にE2Eが大量に壊れる
```

### 改善後の構成

```
E2E: 3-4個（API結合確認のみ）
  - 貸出成功 (201)
  - 上限超過エラー (409)
  - 存在しないリソース (404)

ユニット: 25-30個（全境界値）
  - canBorrow: GENERAL 0〜6冊
  - canBorrow: PREMIUM 0〜16冊
  - getLoanPeriodDays: GENERAL=14, PREMIUM=30
  - calculateDueDate: 各種日付
→ ピラミッド型、仕様変更時はユニットのみ修正
```

## キーメッセージ

**テストピラミッドを守る**
- 境界値・エッジケースはユニットテストで検証する（高速、明確、修正容易）
- E2Eテストは「結合の確認」に限定する（少数だが必要）
- 同じロジックを複数の層でテストすると、メンテナンスコストが倍増する

**E2Eテスト過多の3つの弊害**
1. **実行時間**: ユニットテストの数百倍遅い → CI/CD全体が遅くなる
2. **メンテナンス**: 仕様変更時にユニット+E2Eの両方を修正 → 修正工数が倍
3. **デバッグ**: `Expected 409 but was 201`では原因が分からない → 調査時間が増大

**AI駆動開発での注意**
- AIに「テストを書いて」と依頼するとE2Eを書きがち
- テストの層を明示的に指定する:「ユニットテストとして書いて」
- テストピラミッドの判断は開発者が行う
