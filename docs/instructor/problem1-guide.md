# 課題1: テストの量より質 - 講師向けファシリテーションガイド

> **この課題のゴール:**
> 受講者が「AIに任せればテストも安心」という思い込みから脱却し、テストの品質を自ら評価・改善できるようになること。

---

## この課題の設計意図

この課題は**2段階アプローチ + カバレッジ95%目標**により、確実に「無駄なテスト」を生成させる設計になっています。

### なぜ2段階なのか

1. **第1段階（1-2）**: 「主要なテストケース」のみを指示
   - この段階では、AIは通常60-70%程度のカバレッジのテストを生成
   - 比較的質の良いテストが生成されることが多い

2. **第2段階（1-4）**: 「追加した機能のカバレッジ95%達成」を指示
   - **ここが罠**: 「カバーされていないコード行を実行する」ことが目的化
   - ビジネスロジックの検証ではなく、コード実行が目的になる
   - この段階で `assertNotNull()`, `assertDoesNotThrow()` が量産される

**重要:** カバレッジの対象は追加した延滞料金計算機能のクラスのみです。プロジェクト全体ではなく、受講者が実装したクラス（例：`OverdueFeeCalculator`）のカバレッジを95%にすることを目指します。

### なぜ95%なのか

- **90%**: 良いテストを書いても達成できる。無駄なテストが生成されない可能性がある
- **95%**: かなり厳しいが不可能ではない。この辺りから「無駄なテスト」が増え始める
- **100%**: やりすぎ感があり、指示として不自然

95%は「現実的だが厳しい」という絶妙なラインで、受講者に違和感を与えずに罠にはめることができます。

## 想定されるAI生成テストのパターン

**第1段階（主要テスト）**では比較的良質なテストが生成されることもありますが、**第2段階（カバレッジ95%達成のための追加）**で高い確率で以下のようなテストが生成されます。

### パターン1: 存在確認のみ（第2段階で頻出）

```java
@Test
void testCalculateOverdueFee_generalMember() {
    // ... setup ...
    OverdueFee fee = calculator.calculate(generalMember, lendingRecord);

    assertNotNull(fee);
    assertTrue(fee.amount() >= 0);
}
```

**問題:** `amount()` が 0 でも 999999 でもパスする。ビジネスルールの正しさを一切検証していない。

### パターン2: 型・形式の検証のみ

```java
@Test
void testOverdueFee_returnsValidResult() {
    OverdueFee fee = calculator.calculate(member, record);

    assertNotNull(fee);
    assertInstanceOf(OverdueFee.class, fee);
    assertFalse(fee.toString().isEmpty());
}
```

**問題:** コンパイルが通る時点で型は保証されている。テストとして何の価値もない。

### パターン3: 例外が出ないことの確認のみ

```java
@Test
void testCalculateOverdueFee_doesNotThrowException() {
    assertDoesNotThrow(() -> {
        calculator.calculate(generalMember, lendingRecord);
    });
}
```

**問題:** 実装が `return new OverdueFee(0)` のようなスタブでもパスする。

### パターン4: 一見正しく見えるが甘いアサーション

```java
@Test
void testOverdueFee_generalMember_isPositive() {
    // 5 days overdue
    OverdueFee fee = calculator.calculate(generalMember, fiveDaysOverdue);

    assertTrue(fee.amount() > 0);       // 1でも100000でもパス
    assertTrue(fee.amount() <= 1000);    // 上限以下であることだけ確認
}
```

**問題:** `250` であるべきところを `1` でも `999` でもパスする。

### 理想的なテスト（ステップ3で受講者に書いてほしい形）

```java
@Test
void 一般会員_5日延滞_250円() {
    LocalDate dueDate = LocalDate.of(2025, 5, 1);
    LocalDate returnDate = LocalDate.of(2025, 5, 6);

    OverdueFee fee = calculator.calculate(
        MemberType.GENERAL, dueDate, returnDate);

    assertEquals(250, fee.amount());
}

@Test
void 一般会員_40日延滞_上限1000円() {
    LocalDate dueDate = LocalDate.of(2025, 5, 1);
    LocalDate returnDate = LocalDate.of(2025, 6, 10);

    OverdueFee fee = calculator.calculate(
        MemberType.GENERAL, dueDate, returnDate);

    assertEquals(1000, fee.amount());
}

@Test
void プレミアム会員_10日延滞_200円() {
    LocalDate dueDate = LocalDate.of(2025, 5, 1);
    LocalDate returnDate = LocalDate.of(2025, 5, 11);

    OverdueFee fee = calculator.calculate(
        MemberType.PREMIUM, dueDate, returnDate);

    assertEquals(200, fee.amount());
}

@Test
void 期限当日返却_延滞料金なし() {
    LocalDate dueDate = LocalDate.of(2025, 5, 1);
    LocalDate returnDate = LocalDate.of(2025, 5, 1);

    OverdueFee fee = calculator.calculate(
        MemberType.GENERAL, dueDate, returnDate);

    assertEquals(0, fee.amount());
}
```

## バグ注入シナリオと問いかけ

受講者のステップ1完了後、以下の順番で問いかけてバグを注入させてください。**1つずつ順番に実施**し、毎回テストを実行させます。

### シナリオ1: 日額料金の改ざん（最もインパクトが大きい）

> **講師のセリフ:**
> 「実装コードで、一般会員の日額料金を `50` から `500` に変更してみてください。テストコードは変更しないでください。テストを実行して、結果を確認しましょう。」

**期待される結果:** テストがすべてパスしてしまう

> **問いかけ:**
> 「テストは全部パスしましたね。でも、一般会員の延滞料金が10倍になっています。5日遅れたら2,500円請求されることになりますが、それは正しいですか？」
>
> 「では、このテストは何を保証していたのでしょうか？」

**ポイント:** 受講者に自分のテストコードのアサーション部分を確認させる。`assertNotNull` や `assertTrue(fee >= 0)` のようなアサーションしかないことに気づかせる。

### シナリオ2: 上限金額の削除

> **講師のセリフ:**
> 「次に、延滞料金の上限チェック（cap）のロジックを削除またはコメントアウトしてみてください。上限なしで計算されるようにしてください。」

**期待される結果:** テストがすべてパスしてしまう

> **問いかけ:**
> 「100日延滞したら一般会員は5,000円請求されることになりますが、本来は1,000円が上限のはずですね。テストはこれを検出できていますか？」

### シナリオ3: 境界条件の破壊

> **講師のセリフ:**
> 「最後に、延滞日数の計算で、貸出期限当日を延滞1日目としてカウントするように変更してみてください。つまり、期限ぴったりに返しても1日分の延滞料金が発生するバグです。」

**期待される結果:** テストがすべてパスしてしまう（または、ここだけは落ちるケースもある）

> **問いかけ:**
> 「期限通りに返したのに延滞料金を取られたら、利用者はどう思いますか？これは最もクレームにつながるバグですが、テストで守れていましたか？」

## ファシリテーションの注意点

### タイミングと進め方

| タイミング | 講師のアクション |
|-----------|-----------------|
| ステップ1-3（第1段階完了）| 「追加した延滞料金計算機能のカバレッジはどのくらいですか？」と確認。多くは60-70%程度のはず |
| ステップ1-4（95%達成指示前） | 「では追加した機能のカバレッジ95%を目指してみましょう」と、あくまで自然に誘導 |
| ステップ1-4（95%達成後） | 「カバレッジ95%達成おめでとうございます！実装に自信がありますか？」と確認。ほとんどの受講者が「はい」と答えるはず |
| シナリオ1の前 | 「では、テストが本当に機能しているか確かめてみましょう」と切り出す |
| シナリオ1のテスト実行後 | 十分に間を取る。受講者の「えっ？カバレッジ95%なのに？」という反応を待つ |
| シナリオ2, 3 | 1つ目で衝撃を受けた受講者に追い打ちをかける。パターン的にダメなことを印象づける |
| ステップ3に入る前 | 「では、どうすれば良かったのか？」「カバレッジ95%は何を保証していたのか？」を受講者自身に考えさせてからヒントを出す |

**注意:** 
- カバレッジは追加した延滞料金計算機能（実装したクラス）のみを対象とします。プロジェクト全体のカバレッジではありません。
- OpenShift DevSpaces環境では、HTMLレポートを表示するためにHTTPサーバーを起動します。ポートフォワーディングの通知を見逃した受講者には、VS Codeの「ポート」タブから38080番ポートを探してブラウザで開くよう案内してください。

### カバレッジ戦略が効かなかった場合

2段階アプローチ + 95%目標により、ほとんどの受講者で「無駄なテスト」が生成されるはずですが、まれに例外があります。

- **第1段階で既に95%達成している場合:** 
  - 「素晴らしいですね。では各テストのアサーションを見てみましょう。具体的な期待値（250円、1000円等）を検証していますか？」と質の観点に切り替える
  - それでも良質なテストの場合、次のバグ注入シナリオで問題が明らかになるはず
  
- **第2段階の追加テストも質が高い場合（極めてまれ）:**
  - 「素晴らしいですね。ではPBIの計算例と照らし合わせて、すべてのACがテストでカバーされているか確認してください」と、網羅性の観点に切り替える
  - 他の受講者のテストと比較してもらい、AIの出力は一定でないことを理解させる
  - 「AIがたまたま良いテストを書いたとしても、それが良いテストかどうかを判断できるのは開発者です」と伝える

- **カバレッジが95%に届かない場合:**
  - JaCoCoレポートで実装したクラスを開き、どの行がカバーされていないか確認
  - 実装に問題がある可能性（デッドコード、到達不能なブランチ等）を確認
  - 「95%に届かない理由を調べてみましょう。カバーされていない行は何ですか？実装コードに問題はありませんか？」と実装レビューに誘導
  - それでも届かない場合、「95%は目安です。重要なのはビジネスロジックが正しく検証されているかです」とフォロー

### OpenShift DevSpacesでのトラブルシューティング

受講者から以下のような質問が出た場合の対応：

**「ポートの通知が表示されません」**
- VS Codeの下部パネルで「ポート」タブを開く
- 38080番ポートが表示されているか確認
- 表示されている場合、行の右側にある「ブラウザで開く」アイコン（地球儀マーク）をクリック
- 表示されていない場合、HTTPサーバーが起動しているか確認（`ps aux | grep python3`）

**「ブラウザで開いたが404エラーが出る」**
- `cd target/site/jacoco` を実行してからHTTPサーバーを起動しているか確認
- カレントディレクトリが間違っている可能性がある

**「カバレッジレポートが更新されない」**
- `mvn test jacoco:report` を再実行したか確認
- ブラウザのキャッシュをクリアして再読み込み（Ctrl+Shift+R）
- HTTPサーバーを再起動

**「HTTPサーバーが停止できない」**
- `Ctrl+C` で停止
- それでも停止しない場合、ターミナルを閉じる（新しいターミナルで作業を続けられる）

### 議論を深めるための追加質問

ステップ4の振り返り時に、以下の質問も活用してください。

1. **「カバレッジ95%を達成するために追加したテストは、本当に必要でしたか？」**
   → カバレッジ目標が目的化すると、無駄なテストが増えることを議論

2. **「テストカバレッジ100%のテストスイートがあります。安心ですか？」**
   → カバレッジはコードの実行パスを測るだけで、正しさは測れないことを議論

3. **「カバレッジ指標は何のために使うべきですか？」**
   → テストされていない領域の発見には有用。でも高カバレッジ=品質ではない

4. **「AIにテストを書かせた後、あなたは何をレビューすべきですか？」**
   → アサーションの質、境界値の網羅性、ビジネスルールとの対応

5. **「もしこのバグが本番に出ていたら、どんな影響がありましたか？」**
   → 料金の過大請求、信用失墜、法的リスク等。テストの品質はビジネスリスクに直結する

6. **「AIがテストを書く時代に、開発者のテストスキルは不要になりますか？」**
   → むしろ重要性が増す。AIの出力を評価する能力 = テストの本質を理解する能力

## まとめ（講師が伝えるべきキーメッセージ）

> **「カバレッジは重要な指標ですが、それ自体が目的になってはいけません。AIは"カバレッジを高めるテスト"を大量に書けますが、そのテストがビジネスルールを正しく検証しているかを判断できるのは、ドメインを理解した開発者だけです。」**
>
> **「AI駆動開発の時代だからこそ、テストの本質――何を検証すべきか――を理解するスキルがより重要になります。カバレッジ指標に踊らされず、価値のあるテストを見極める目を養いましょう。」**
